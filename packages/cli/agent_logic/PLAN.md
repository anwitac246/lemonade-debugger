# Lemonade AI Debugger – Architecture Plan

## Goal

Build a **terminal-native AI debugger** that works alongside a normal shell. The CLI is built with **Ink + React** and allows users to run regular shell commands while invoking AI debugging through commands like:

```
lemonade debug <query>
```

The system combines:

* Tool-based agents
* Debug Adapter Protocol (DAP)
* Model routing
* Hierarchical memory
* Safe permission controls
* Reflection and self-critique loops
* LangGraph-based orchestration with checkpointing

The goal is to create a **lightweight terminal-first AI debugging assistant with deep runtime awareness**.

# Agentic Layer

The agent layer is orchestrated using **LangGraph**. Agents communicate through tools and share a central session state object. The graph is compiled with a **Checkpointer** so sessions survive terminal restarts.

---

## Shared State

All agents read from and write to a single shared state. This is the source of truth for the entire debugging session.

Key fields in state:

* **userQuery** – the original debugging request
* **loadedFiles** – files pulled in by the Context Agent
* **analysisResults** – lint/compile output
* **debugSession** – live breakpoints, stack trace, variables
* **proposedPatch** – the fix generated by the Fix Agent
* **pendingPermission** – any action awaiting user approval
* **reflectionScore** – quality score (0–1) of the proposed fix
* **nextAgent** – routing signal for which agent runs next

All tool outputs are typed and validated with **Zod schemas** to prevent agents from misinterpreting results.

---

## Agent Graph Flow

```
          ┌─────────────────────────────────────────────┐
          │                                             │
User ──► Planner ──► Context ──► StaticAnalysis        │
                                      │                 │
                          resolved? ──┤                 │
                               no     │                 │
                                      ▼                 │
                                 Debug Agent            │
                                      │                 │
                                      ▼                 │
                                 Web Agent (optional)   │
                                      │                 │
                                      ▼                 │
                                  Fix Agent             │
                                      │                 │
                                      ▼                 │
                                  Reflect ── score < 0.7 ──┘
                                      │
                                 score >= 0.7
                                      │
                                      ▼
                             [INTERRUPT] User Approval
                                      │
                                      ▼
                               Apply Patch ──► END
```

The **Reflect → Fix loop** is the key difference from a naive linear pipeline. If the fix doesn't score well against the original bug, the graph cycles back automatically before showing anything to the user.

## Planned Agents

### 1. Planner Agent

The entry point. Every debugging session starts here.

Responsibilities:
* Interpret the user's request
* Choose the debugging strategy (static-first vs runtime-first)
* Decide which agent to invoke next

Example: User runs `lemonade debug "appointmentService crashes on null user"`. Planner determines: load service files first, then run static analysis before considering a runtime attach.

Model used: **small model** (classification and routing task, doesn't require deep reasoning)

---

### 2. Context Agent

Collects the relevant code before any analysis begins.

Tools used:
* read_file
* list_files
* search_code (ripgrep-based)
* parse_ast (tree-sitter)

Goal: Load the **minimum viable context**. If the user asks about `appointmentService.ts`, don't load the entire codebase — only that file, its imports, and the functions directly involved.

This keeps token usage low and responses fast.

Model used: **small model**

---

### 3. Static Analysis Node

Not a full LLM agent — this is a **deterministic node** that runs tools and writes results to state.

Tools run:
* run_linter
* run_compiler

Decision point: If errors are found and are self-explanatory (e.g. a type error with a clear fix), the graph routes directly to the Fix Agent, skipping the debugger entirely.

Example: A TypeScript compile error like `Argument of type 'null' is not assignable to parameter of type 'User'` can be fixed without ever attaching a debugger.

---

### 4. Debug Agent

Activates only when static analysis cannot resolve the issue.

Tools used:
* start_debugger
* set_breakpoint
* get_stack_trace
* inspect_variables
* continue_execution

Uses **Debug Adapter Protocol (DAP)** over TCP/stdio. For Node.js this uses `@vscode/debugadapter`. For Python this uses `debugpy`.

Example use case: A function crashes only with specific runtime data that isn't visible from the source. The Debug Agent attaches, sets a breakpoint at the crash site, and inspects the live variable values at that moment.

Model used: **large model** (runtime reasoning requires deeper thinking)

---

### 5. Web Agent

Optional node. Activated when the issue involves external libraries, version mismatches, or undocumented behavior.

Tools used:
* web_search
* search_docs

Example use cases:
* "This error only appears in Express 4.19 — what changed?"
* "The prisma error message suggests a schema migration issue, look up the migration docs"

The Web Agent enriches the Fix Agent's context before patch generation.

Model used: **small model**

---

### 6. Fix Agent

Generates a concrete patch to fix the identified bug.

Responsibilities:
* Generate a minimal diff-style patch (not a full file rewrite)
* Explain the reasoning behind the fix
* Pass the patch and explanation to the Reflect node

Uses patch-based edits. Example:

```
- login()
+ login(user)
```

The Fix Agent never writes to disk directly. It only proposes.

Model used: **large model**

---

### 7. Reflect Node

A self-critique step between Fix Agent and user approval. This prevents bad patches from reaching the user.

The node evaluates:
* Does the patch address the root cause?
* Could the patch introduce regressions?
* Is the explanation coherent with the actual bug?

Produces a score from 0.0 to 1.0.

* Score >= 0.7 → proceed to user approval
* Score < 0.7 → route back to Fix Agent with the critique as additional context

Example: Fix Agent proposes wrapping a null check around a function call, but Reflect notices the real bug is upstream where `user` is never assigned — it sends this critique back so the Fix Agent can revise.

Model used: **large model**

# Tool Layer

All tools are deterministic functions — no LLM calls inside tools. Tools are the only way agents interact with the outside world.

## Filesystem Tools

* read_file – reads a single file's content
* write_file – writes content (requires approval)
* list_files – lists directory contents

## Code Discovery

* search_code – ripgrep-based fast text/regex search across the project

## AST Parsing

* parse_ast – tree-sitter based. Supports Node/TypeScript and Python. Used to extract function signatures, imports, and call graphs without loading full files.

## Static Analysis

* run_linter – runs ESLint / Pylint
* run_compiler – runs TSC / mypy
* parse_ast – see above

## Debugging (DAP)

* start_debugger – launches debug adapter and attaches to process
* set_breakpoint – inserts a breakpoint at a file + line
* get_stack_trace – retrieves the current call stack
* inspect_variables – reads variable values at the current frame
* continue_execution – resumes execution after a pause

## Web / Documentation

* web_search – searches the web for relevant documentation or error explanations
* search_docs – targets framework-specific documentation (e.g. Express, Prisma, React)

## Patch System

* generate_patch – produces a diff-style patch
* preview_patch – formats the patch for display in the terminal UI
* apply_patch – writes the patch to disk (requires user approval)

# Debug Adapter Protocol (DAP)

DAP enables **true runtime debugging** — the ability to pause execution, inspect live state, and step through code. This is what separates Lemonade from tools that only do static analysis.

## Transport

DAP communicates over TCP or stdio. Lemonade implements a thin client that handles:
* DAP message framing (Content-Length headers)
* Request/response matching by sequence number
* Event handling (breakpoint hit, process exit, etc.)

## Runtime Adapters

| Runtime    | DAP Adapter              |
| ---------- | ------------------------ |
| Node.js    | @vscode/debugadapter     |
| Python     | debugpy                  |
| Future     | lldb-dap, delve (Go)     |

## Workflow

```
Static Analysis runs
      ↓
Issue unresolved
      ↓
Debug Agent activates
      ↓
DAP client connects to process
      ↓
Breakpoints inserted at suspect locations
      ↓
Process runs until breakpoint hit
      ↓
Stack trace + variable values collected
      ↓
Agent reasons about live runtime state
      ↓
Findings passed to Fix Agent
```

Example: `appointmentService.ts` line 47 throws `Cannot read property 'id' of null`. Debug Agent sets a breakpoint at line 47, runs the service, and inspects variables at the point of crash — discovering that `currentUser` is null because a middleware was skipped.

# Permission System

Certain actions require explicit user approval before execution. This uses **LangGraph's native `interrupt()` primitive** — the graph pauses mid-execution and waits for a user response before continuing.

| Action           | Permission |
| ---------------- | ---------- |
| read files       | always allowed |
| search code      | always allowed |
| run linter       | always allowed |
| run commands     | ask user |
| start debugger   | ask user |
| write files      | ask user |
| web access       | ask user |

Example terminal prompt:

```
Lemonade wants to run:
  npm test

Allow? (y/n)
```

If the user denies, the graph receives the rejection, notes it in state, and the Planner decides how to proceed without that action.

---

# Model Routing

A routing layer selects the appropriate model for each task to balance cost, speed, and capability.

| Task                  | Model       | Reason                              |
| --------------------- | ----------- | ----------------------------------- |
| Planning / routing    | small model | Simple classification               |
| Context gathering     | small model | Tool calls, no deep reasoning       |
| Debug reasoning       | large model | Complex runtime state analysis      |
| Patch generation      | large model | Requires deep code understanding    |
| Reflection / critique | large model | Nuanced quality evaluation          |
| Web lookup            | small model | Mostly retrieval, minimal reasoning |

# Memory Management

Lemonade implements a **multi-layer memory system** to keep prompts small while maintaining context across agents and sessions.

---

## 1. Short-Term Session Memory

Stores the active debugging conversation as it unfolds.

What gets stored:
* User query
* Files loaded by Context Agent
* Tool execution history
* Agent decisions and reasoning

When a conversation grows too large, it is compressed into a structured snapshot:

```
<state_snapshot>
  goal: debug appointmentService.ts
  key_facts:
    - currentUser is null at line 47
    - auth middleware skipped for this route
  recent_actions:
    - read appointmentService.ts
    - ran compiler (0 errors)
    - attached debugger
  status:
    fix proposed, awaiting approval
</state_snapshot>
```

This snapshot replaces the full history in the prompt, keeping token usage bounded.

---

## 2. Hierarchical Long-Term Memory (LEMONADE.md)

Persistent context stored in Markdown files at two levels:

```
~/.lemonade/LEMONADE.md     ← global user preferences
project_root/LEMONADE.md    ← project-specific rules
```

Global memory stores:
* User's preferred coding style
* Preferred debugging approaches
* Tools to always/never use

Project memory stores:
* Architecture rules
* Framework conventions
* Testing requirements
* Known gotchas

Example project LEMONADE.md:

```
# Project Rules
Controllers must not access the database directly — use services.
All fixes must have a corresponding test.
This project uses Prisma — never write raw SQL.
```

Agents read these files at session start. Users can edit them manually at any time.

---

## 3. Persistent Memory Tool

Agents can write new learnings to memory mid-session:

```
save_memory("User prefers tabs instead of spaces")
save_memory("Auth middleware is opt-in on this project — routes must explicitly include it")
```

This appends to `~/.lemonade/LEMONADE.md` and persists across all future sessions.

---

## 4. Debug Session Memory

Live runtime state stored separately from the conversation history.

What gets stored:
* Active breakpoints and their locations
* Stack traces captured
* Variable values at each breakpoint
* Runtime environment details (Node version, env vars, etc.)

This state is passed directly into the Debug Agent's context so it can reason about actual execution rather than guessing from source code alone.

---

## 5. LangGraph Checkpointer

The graph state is persisted using LangGraph's checkpointing system.

In development: **MemorySaver** (in-memory, lost on restart)
In production: **SQLite or Postgres checkpointer** (survives restarts)

This means if a terminal session is interrupted mid-debug, running `lemonade resume` restores the exact graph state and continues where it left off.

# Auto-Fix Workflow

Fixing code uses a deliberate pipeline — never an immediate overwrite.

```
Analyze bug (static + runtime evidence)
      ↓
Generate patch (Fix Agent)
      ↓
Validate patch (Reflect node, score >= 0.7)
      ↓
Preview patch shown to user in terminal
      ↓
User approves (interrupt)
      ↓
Apply patch to disk
```

The user always sees a diff-style preview before anything is written:

```
appointmentService.ts line 47:

- const appointment = await db.create(currentUser.id, data)
+ if (!currentUser) throw new Error("No authenticated user")
+ const appointment = await db.create(currentUser.id, data)
```

# Error Recovery

The graph has defined fallback behavior for tool and agent failures — not silent errors.

Failure scenarios and recovery:

| Failure                     | Recovery Strategy                                    |
| --------------------------- | ---------------------------------------------------- |
| Debugger fails to attach    | Fall back to static analysis only, inform user       |
| Linter times out            | Skip linting, continue to debug node                 |
| Web search returns nothing  | Proceed without external docs, note in explanation   |
| Fix Agent produces bad patch| Reflect node catches it, routes back for revision    |
| User denies all permissions | Planner re-routes to read-only analysis path         |

Each node has a defined error output edge in the LangGraph graph definition.

# Observability

**LangSmith tracing is enabled from day one.**

Every agent invocation, tool call, and graph transition is traced. This is essential for debugging the debugger during development.

What gets traced:
* Which agent ran and for how long
* Every tool call with inputs and outputs
* Model calls with token counts
* Graph routing decisions
* Reflection scores over time

In production, traces help identify which steps are slow, expensive, or producing poor results.

---

# Performance Strategy

Lemonade is designed to feel fast in a terminal environment.

Strategies:

* **Lazy file loading** – Context Agent loads files on demand, not upfront
* **Ripgrep for search** – Fast native search instead of walking the filesystem in JS
* **Small context windows** – Only relevant files/functions enter the prompt
* **Tool output caching** – Linter and compiler results cached within a session
* **Streaming LLM responses** – Output streams to terminal as it's generated, no waiting
* **Agent step limits** – Each agent has a max step count to prevent infinite loops
* **Model routing** – Cheap models for cheap tasks

# Full Debugging Workflow (Example)

User runs:

```
lemonade debug "appointmentService crashes when creating appointments"
```

Step by step:

1. **Planner** interprets the request. Decides to load the service file first, then run static checks.
2. **Context Agent** loads `appointmentService.ts`, its imports, and the related auth middleware.
3. **Static Analysis** runs the TypeScript compiler and linter. No errors found — the crash is runtime-only.
4. **Planner** routes to Debug Agent. Permission prompt appears: "Lemonade wants to attach a debugger. Allow? (y/n)"
5. **Debug Agent** attaches via DAP, sets breakpoint at the crash site, runs the service.
6. Breakpoint hits. Stack trace shows: `createAppointment → authMiddleware → (skipped)`. Variable `currentUser` is `null`.
7. **Web Agent** is skipped (issue is clear from runtime state alone).
8. **Fix Agent** generates a null-check patch with explanation.
9. **Reflect node** scores the patch 0.85. Passes.
10. Terminal shows diff preview. User approves.
11. Patch applied to disk.

Total time: under 60 seconds for a runtime bug that would take a developer 10–20 minutes to track down manually.

# Build Order

Given the Ink + React frontend is complete, build in this sequence:

1. **Tool layer** – filesystem, linter, compiler tools with Zod validation
2. **DAP client + Node.js adapter** – get breakpoints and stack traces working in complete isolation before integrating into the graph
3. **LangGraph state schema + basic Planner → Context → Fix flow** – no debugger yet
4. **Wire in Static Analysis node** with conditional routing
5. **Wire in Debug Agent + DAP** as the runtime path
6. **Add Reflect node** and the fix loop
7. **Add permission interrupts** using LangGraph's interrupt primitive
8. **Memory layer** – LEMONADE.md reading + MemorySaver checkpointer
9. **Connect Ink frontend to graph via streaming**
10. **LangSmith tracing setup**

The biggest technical risk is the **DAP transport layer**. Validate it standalone before integrating with the graph.
